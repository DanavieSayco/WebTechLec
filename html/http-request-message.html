Request Line

The Request-Line begins with a method token/request method, followed by the Request-URI and the protocol version, and ending with CRLF (terminated line consisting of three spaced-separated items).
  • Request Methods
    •
    It passes the information about the request and the client to the server
    • GET
      • Transfer a current selected representation of the resource identified by the request URI
      • Requests data from a specified resource
      • Most commonly used method
      • Must be supported by all compliant general-purpose servers
      • Can be stored
      • Can be bookmarked
      • Must never be utilized when managing delicate information
      • Are ought to be utilized just to recover information
      • Asks for have length limitations
      • Remains in the browser history
    • HEAD
      • Same as GET but the entity is not sent
      • Used to retrieve metadata about the entity
      • Must also be supported by all
    • POST
      • Performing resource-specific processing of entities enclosed in the message body
      • Submits data to be processed to a specified resource
      • Has a payload
      • Are never cached
      • Cannot be bookmarked
      • Do not remain in the browser history
      • No restrictions on the data length
    • PUT
      • Store the enclosed entity in the message body under a specified URI
      • By default, the method is not allowed
      • When allowed, authentication must be provided
      • Uploads a representation of the specified URI
    • DELETE
      • Remove the resource associated with the specified URI
      • Like PUT, the method is not allowed by default
    • OPTIONS
      • Request information on what can be done with the resource specified
      • Returns the HTTP methods that the server supports
    • TRACE
      • Perform or request a loopback of the requested message (echo back)
      • Typically used for testing/diagnosis of the request/response chain
      • Allows the customer to perceive what is being gotten at the flip side of the demand chain and use the data for testing
    • CONNECT
      • Establishment of an encryption tunnel to communicate with https
      • Converts the request connection to a transparent TCP/IP tunnel
    • Extension Methods
      WebDAV
      • PROPFIND - method recovers properties characterized on the asset recognized by the Request-URI
      • PROPPATCH – method forms guidelines indicated in the demand body to set and additionally or expel properties characterized on the asset recognized by the Demand URI
      • MKCOL – make collection (new folder)
      • COPY – creates a duplicate of the source resource
      • MOVE – does not guarantee the ability to move a resource to a particular destination
      • LOCK - lock resources
      • UNLOCK – unlock resources
    • Safe Methods
      • Doesn’t change the resources
      • Method that can be cached
      • GET, HEAD, OPTIONS, TRACE
    • Idempotent Methods
      • Repeated request result to the same response
      • GET, HEAD, OPTIONS, TRACE, PUT, DELETE
    • Cacheable Methods
      • Requests that generate cache
      • GET, HEAD are cacheable
  • Request URI
    •
    Request-URI = "*" | absoluteURI | abs_path | authority
    The Request URI, which URI means 'Uniform Resource Identifier', identifies the resource upon which to apply the request.
    • *

      The asterisk * form is used when an HTTP request does not apply to a particular resource, but to the server itself, and is only allowed when the method used does not necessarily apply to a resource.

    • absoluteURI

      The absoluteURI is used when an HTTP request is being made to a proxy. The proxy is requested to forward the request or service from a valid cache, and return the response.

    • abs_path

      The abs_path form is used to identify a resource on an origin server or gateway. This cannot be empty; if the original URI is empty, it MUST be given as "/" (the server root).

    • authority

      The authority form is only used by the CONNECT method (method def. in Request Methods - CONNECT.)

  • HTTP Protocol Version
    •
    • A server supporting HTTP version 1.1 will return the version information. (e.g. HTTP-Version = HTTP/1.1)

Request Header Fields

Request Header Fields provide information about the request messages requested by the client.
  • Accept
    •

    The Accept request header field specifies acceptable file types. If other types can be delivered then it will be accepted. The server responds with negotiation.

  • Accept-Charset
    •

    The Accept-Charset request header field indicates which character sets are acceptable for the response.

  • Accept-Encoding
    •

    The Accept-Encoding request header field restricts the content-codings that are acceptable in the response.

  • Accept-Language
    •

    The Accept-Language request header field restricts the set of languages that are preferred as a response to the request.

  • Authorization

    • The Authorization request header field ensures protection from unauthorized access. When proper credentials are entered, another request will be sent with authorization.

  • Proxy Authorization
    •

    The Proxy Authorization request header field allows the client to identify itself (or its user) to a proxy which requires authentication.

  • Expect
    •

    The Expect request header field indicates that particular server behaviors are required by the client.

  • From
    •

    The From request header field contains contact credentials of the sender

  • Host
    •

    The Host request header field specifies the internet host and port number of the resource being requested. This enables the origin server to distinguish among resources while servicing requests for multiple host names on a single IP address.

  • If-Match
    •

    The If-Match request header field compares the file in the cache and server to check if file is unmodified.

  • If-None-Match
    •

    The If-None-Match request header field requests the server to perform the requested method only if one of the given value in a tag matches the given entity tags. This is also used to update caches or to prevent to upload a new resource when one is already existing.

  • If-Range
    •

    The If-Range request header field allows the client to “short circuit” the second request. It can be used with a conditional GET to request only the portion of the entity that is missing.

  • If-Unmodified-Since
    •

    The If-Unmodified-Since request header field expects the entity to be transmitted only if it has not been modified after the given date.

  • Ranges
    •

    The Ranges request header field specifies portions of the resource the client wants to receive.

  • Max-Forwards
    •

    The Max-Forwards request header field limit the number of proxies or gateways that can forward the request.

  • Referer
    •

    The Referer request header field specifies the URI where the request was originated. It is also used to check where traffic is coming from.

  • TE
    •

    The Trailer Encoding request header field specifies the transfer encodings the user agent is willing to accept.

  • User-Agent
    •

    The User-Agent request header field contains additional information about the clients' identity.

Request Message

Example of a Request Message body:

GET /sample.html HTTP/1.1
User-Agent: Mozilla/4.0 (compatible; MSIE5.01; Windows NT)
Host: www.sample.com
Accept-Language: en-us
Accept-Encoding: gzip, deflate
Connection: Keep-Alive
